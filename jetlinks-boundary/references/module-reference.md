# JetLinks 模块边界与依赖决策指南

本文件解决的核心问题不是“有哪些模块”，而是“当一个功能需要使用另一个能力时，应该如何穿越边界”。

## 何时使用

- 不确定该直接依赖另一个模块，还是走命令服务
- 不确定该在保存后直接处理，还是发事件
- 不确定该做请求式调用，还是订阅式消费

## 四种常见边界方式

### 1. 直接依赖

适用：
- 同一部署单元内的稳定内部能力
- 编译期强依赖合理
- 目标能力本来就作为本地扩展点被复用

不适用：
- 跨服务边界
- 明显需要路由、权限继承、远程调用抽象
- 未来希望独立部署的能力

### 2. 命令服务

适用：
- 查询或触发另一个边界内的能力
- 当前脚手架已经使用命令模式或服务代理
- 需要统一权限、路由、协议或可替换实现

不适用：
- 当前工作区没有命令服务体系
- 本地内部扩展点已经明确用直接注入

### 3. 事件驱动

适用：
- 保存、修改、删除后的副作用
- 松耦合联动
- 一个动作需要触发多个后续处理

不适用：
- 用户请求需要立刻拿到结果的同步查询
- 需要明确事务内返回值的场景

### 4. 实时订阅

适用：
- Topic、消息流、持续消费
- 系统广播、设备消息、跨节点传播

不适用：
- 一次性查询
- 明确的同步命令执行

## 决策顺序

1. 先看当前脚手架已经公开了什么边界
   - 查命令服务、事件类、订阅处理器、公共接口模块。

2. 再看目标能力属于哪一类
   - 同步查询、远程操作、保存后联动、连续消息流。

3. 最后选择最小耦合方案
   - 不额外引入新的边界机制。

## 低上下文模板仓库默认决策

如果当前仓库没有足够参考实现，按以下优先级决策：

1. CRUD 后的副作用，默认优先事件驱动
   - 不把同步联动、清理、广播都塞进控制器或 Service 主流程。

2. 同一脚手架内的同步能力复用，默认优先直接依赖
   - 前提是当前仓库尚未建立命令调用基础设施。

3. 只有在以下情况才优先命令服务
   - 当前依赖和目录结构已经表明存在命令框架
   - 或用户明确要求建立命令/RPC 边界

4. 持续流式消费或广播，默认优先订阅
   - 不把消息流场景硬做成同步查询。

5. 如果采用的是“新建约定”而不是“复用约定”
   - 必须在分析或总结中明确说明这是模板仓库下的默认决策。

## 发现要点

### 判断是否该直接依赖

检查：
- 相邻模块是否直接注入同类服务
- 是否存在公共接口或基础抽象供本地扩展
- 目标能力是否与当前模块处于同一业务边界

### 判断是否该走命令服务

检查：
- 是否已存在 `@CommandService` 或服务代理
- 是否已有固定 service/support id
- 当前模块是否已有同类 consumer 示例

如果没有现成示例但用户明确要求命令边界：
- 先确认依赖中已经具备命令框架
- 再用最小命令层落地，不同时发明多套命名规则

### 判断是否该发事件

检查：
- 当前模块是否已有 `@EventListener` 处理实体生命周期事件
- 是否已有领域事件类
- 副作用是否应在事务提交后执行

### 判断是否该做订阅

检查：
- 当前工作区是否已有 `@Subscribe`、EventBus 或 Topic 约定
- 是否已有同类消息源的 Topic 模式

## 禁止事项

- 不要臆造 service id、command id、topic、事件名。
- 不要在已有命令边界的场景下直接跨模块注入实现类。
- 不要为了一个保存后副作用，把控制器或服务写成巨型流程编排器。

在低上下文模板仓库中同样禁止：

- 因为“以后可能会用到”就默认引入命令边界
- 因为缺少事件类就把所有副作用写进 CRUD 主流程

## 输出模板

分析时输出：

1. 目标能力所在边界
2. 当前脚手架已有的可用穿越方式
3. 推荐方案
4. 不推荐方案及原因
