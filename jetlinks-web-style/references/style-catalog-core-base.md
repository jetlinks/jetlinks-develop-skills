## 反传统后台感硬约束

在挑选任何方案前，先确认下面这些约束。命中即必须改写当前默认实现，避免页面默默回退成"后台管理 + 大表格 + 全字段弹窗"的传统套壳。

### 1. 编辑触发梯度（按业务最小代价优先）

编辑触发方式必须按下面顺序优先评估，能用前者就不要用后者：

1. `inline 单字段编辑`：`InputEditable` / `Editable` / `FormItemEditable` 直接在原位触发
2. `分区局部编辑`：`SectionCard` / `KvGrid` 内部分区切换编辑态，配 `StickyActionBar`
3. `JlDrawerShell 局部抽屉`：单一目标、字段较多、希望保留上下文时
4. `EditDialog 完整弹窗`：字段多、强校验、跨字段依赖
5. `独立配置页`：多阶段、长流程、需要独立 URL

> 对名称、标签、说明、备注、描述、状态切换、单一开关这类轻量字段，**禁止默认打开完整编辑弹窗**。

### 2. 首屏不可叠加

下列叠加在没有业务证据时禁选，命中至少一条且无法证明数据来源 / 决策价值 / 动作承接时，必须删除或降级：

- "4 KPI 卡 + 顶部搜索栏 + 大表格"的经营分析三段套壳
- 顶部 banner + 装饰插画 + 介绍卡 + 表格的"营销首页"套壳
- 多 Tab 平铺但每个 Tab 只是同类字段的另一种排列
- 没有真实数据源、刷新机制、动作承接的统计数字 / 趋势图 / 排行榜
- 没有业务对应关系的"快捷入口墙"

**一屏一主视觉锚点（硬规则）**：主 CTA、关键数据带、真实产品状态区，**三选一，不叠加**。一屏内出现 2 个及以上等权重的强视觉块（彩色 hero / 大数字 KPI / 主操作按钮区）就会争抢焦点，长时间停留视觉疲劳。具体落地：

- 工作流页主要内容（表格 / 列表 / 详情）必须**首屏可见**；不允许用解释型 hero、装饰卡片、教程文案把它推下去
- 一个 KPI 区域不要同时和"大尺寸主 CTA + 突出 hero + 强烈状态条"并排出现；选其中一个作为主锚点
- 工作台 / dashboard 的 CTA 区允许**一处**单色淡渐变锚点（透明度 ≤ 12% / 单色或同色系 / 不带 blob / 不带光晕 / 不带大引号）；其余位置不重复

### 3. 状态先于字段

列表 / 卡片 / 摘要区的第一信息层必须是状态、标签、归属、主动作，而不是把所有字段平铺：

- 状态用 `j-badge-status` / `MetaChip` / `AppTag`，不要用裸文字
- 主动作就近放在卡片或行内，不堆 toolbar
- 字段密度让位给状态识别度

**状态强调收口（硬规则）**：

- 风险、告警、选中、处理中等状态优先用 `.pill` / 小 dot / **完整描边** / 弱状态底 / 行内状态列
- **禁止**`border-left: 3px solid ...` / `box-shadow: inset 4px 0 0 ...` 这类**单侧彩色线条**作为主要状态信号（旧后台风格，且破坏整站克制配色体系）
- 业务列表已有 status pill 时，行本身只需 hover / selected 弱底 + 完整描边，**不再叠彩色左条**
- 局部组件需要新类型色 → 先在 token 层补域 token，再由组件引用，不在 store 或组件级 CSS 里写散落 hex / rgb()

### 4. 信息密度目标

- 卡片墙：每行 3-4 张，单卡核心字段 ≤ 5 + 一个状态徽标 + 1-2 个主动作
- 摘要列表：单行只表达一个对象的状态、标签、归属与主动作，长字段截断 + tooltip
- 标准表格：列数控制在 6-10 列，操作列只放 1-2 个常用动作，其余进入"更多"
- 详情分区：分区内字段 ≤ 8，超出就拆区或拆 Tab

### 5. 文案面向终端用户

#### 先识别功能的最终用户

在写任何用户可见文案前（功能介绍、说明卡、引导提示、空态、错态、Tooltip、按钮、Tab、占位、校验消息、Popconfirm、弹窗标题与说明、表单字段描述、提交反馈、帮助入口），先回答：

- 这个功能的最终用户是谁（运营 / 运维 / 平台管理员 / 租户管理员 / 设备管理员 / 业务管理员 / 终端使用者 / 开发集成方等具体角色）
- 用户进入页面想要完成的目标是什么
- 用户能看懂的术语是什么（不是字段 key、变量名、内部 API 路径、接口字段、组件名）
- 用户的下一步动作是什么

回答不上来时，先问，不要硬写。

#### 不允许出现

- "待接接口"、"这里以后接数据"、"模块说明"、"交互方式"、"设计意图"、"原型说明"、"占位文案"、"TODO"、"待定" 等开发态 / 原型态文字
- 用户的**原始需求文本、prompt、内部任务描述、需求文档原文、开发故事卡描述**被直接放进页面文案、说明卡、Tooltip、空态提示、帮助区域或弹窗描述
- 开发视角的描述："本功能用于实现...""我们将提供...""这里展示用户希望的...""按需求实现 XX 能力"
- 字段 key、变量名、API 路径、组件名、Topic 名、命令 ID、权限码、内部 schema 名出现在用户可见文案中
- "操作成功"这类没有上下文的空泛提示；改为带对象名 / 影响范围 / 下一步的具体反馈，例如"已停用网关 A，关联的 5 台设备将在 1 分钟内下线"
- 字段展示名、列头、按钮、状态文案硬编码到代码中（统一走当前 i18n 入口）
- 把同一段"功能简介"复制贴到首屏 banner、空态、Tooltip、帮助入口里，造成信息冗余

#### 必须做到

- 先从最终用户视角改写：用动作动词开头（如"启用 / 停用 / 接入 / 配置 / 查看 / 处置"），少用"用于 / 旨在 / 通过 / 借助"等需求文档口吻
- 一句话能说清就不写两句；功能介绍 ≤ 2 行能讲清"它能做什么 + 启用的前置条件 + 下一步去哪"
- 状态、空态、错态、加载态文案具体到对象与下一步动作，例如"暂未接入设备，先创建一个产品再添加设备"，而不是"暂无数据"
- 校验文案给具体限制和示例，例如"长度限制 32 字符内，仅支持字母、数字、`_`、`-`"，而不是"输入有误"
- 涉及多角色时，文案按角色就近呈现（管理员可见的文案不应出现在租户视图）
- 帮助 / 说明优先以 Tooltip、Popconfirm、空态引导、状态附加说明的方式承接，而不是顶部塞一块说明 banner

### 6. 弹窗不是唯一编辑路径

详情页、配置页、对象工作区必须先评估能否走 inline / sectional 编辑，再决定是否需要弹窗。如果整个页面所有编辑动作都进入同一个 EditDialog，必须给出业务理由（强校验 / 跨字段依赖 / 阶段化提交），否则视为反模式。
