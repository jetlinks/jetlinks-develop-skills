# 页面交互方案目录

本文件是 JetLinks 前端 **页面交互方案的扩展目录**。用于在用户需要选择页面壳层、信息架构与交互骨架时提供候选方案；最终实现仍必须回到 `jetlinks-web`，按当前 workspace 的 `jetlinks-web-core` 核验组件、hooks、utils 与相邻页面用法。

## 反传统后台感硬约束

在挑选任何方案前，先确认下面这些约束。命中即必须改写当前默认实现，避免页面默默回退成"后台管理 + 大表格 + 全字段弹窗"的传统套壳。

### 1. 编辑触发梯度（按业务最小代价优先）

编辑触发方式必须按下面顺序优先评估，能用前者就不要用后者：

1. `inline 单字段编辑`：`InputEditable` / `Editable` / `FormItemEditable` 直接在原位触发
2. `分区局部编辑`：`SectionCard` / `KvGrid` 内部分区切换编辑态，配 `StickyActionBar`
3. `JlDrawerShell 局部抽屉`：单一目标、字段较多、希望保留上下文时
4. `EditDialog 完整弹窗`：字段多、强校验、跨字段依赖
5. `独立配置页`：多阶段、长流程、需要独立 URL

> 对名称、标签、说明、备注、描述、状态切换、单一开关这类轻量字段，**禁止默认打开完整编辑弹窗**。

### 2. 首屏不可叠加

下列叠加在没有业务证据时禁选，命中至少一条且无法证明数据来源 / 决策价值 / 动作承接时，必须删除或降级：

- "4 KPI 卡 + 顶部搜索栏 + 大表格"的经营分析三段套壳
- 顶部 banner + 装饰插画 + 介绍卡 + 表格的"营销首页"套壳
- 多 Tab 平铺但每个 Tab 只是同类字段的另一种排列
- 没有真实数据源、刷新机制、动作承接的统计数字 / 趋势图 / 排行榜
- 没有业务对应关系的"快捷入口墙"

### 3. 状态先于字段

列表 / 卡片 / 摘要区的第一信息层必须是状态、标签、归属、主动作，而不是把所有字段平铺：

- 状态用 `j-badge-status` / `MetaChip` / `AppTag`，不要用裸文字
- 主动作就近放在卡片或行内，不堆 toolbar
- 字段密度让位给状态识别度

### 4. 信息密度目标

- 卡片墙：每行 3-4 张，单卡核心字段 ≤ 5 + 一个状态徽标 + 1-2 个主动作
- 摘要列表：单行只表达一个对象的状态、标签、归属与主动作，长字段截断 + tooltip
- 标准表格：列数控制在 6-10 列，操作列只放 1-2 个常用动作，其余进入"更多"
- 详情分区：分区内字段 ≤ 8，超出就拆区或拆 Tab

### 5. 文案面向终端用户

不允许出现：

- "待接接口"、"这里以后接数据"、"模块说明"、"交互方式"、"设计意图"
- "操作成功"这类没有上下文的空泛提示，改为带对象名 / 影响范围的具体反馈
- 字段展示名、列头、按钮、状态文案硬编码到代码中（统一走当前 i18n 入口）

### 6. 弹窗不是唯一编辑路径

详情页、配置页、对象工作区必须先评估能否走 inline / sectional 编辑，再决定是否需要弹窗。如果整个页面所有编辑动作都进入同一个 EditDialog，必须给出业务理由（强校验 / 跨字段依赖 / 阶段化提交），否则视为反模式。

### 7. 详情页头部摘要区：基本信息直接 inline 编辑

所有承载"单对象详情"的方案（对象详情工作区 / 主从详情工作区 / 配置态-运行态切换页，及任何带"对象摘要"的派生方案），头部摘要区必须按下面规范实现，保持跨页面一致。

#### 必须包含的要素（按顺序）

1. **对象身份**：图标 + 名称（**inline 可改**），名称是强主标题
2. **状态**：状态徽标 / chip（用 `j-badge-status` / `MetaChip`，不用裸文字；状态切换通过快捷动作按钮，不放进编辑表单里）
3. **标签**：胶囊集合（用 `ChipGroup` / `AppTag`），**就近支持 + / × 增删**，禁止进编辑表单改标签
4. **说明 / 描述**：副标题位（**inline 可改**），长文本截断 + tooltip 或"展开 / 收起"
5. **归属**：项目 / 租户 / 责任人 / 区域 / 产品等（图标 + 文字，必要时可跳转），通常只读
6. **元信息**：创建时间 / 更新时间 / 版本号（次要文字，弱化展示）
7. **快捷动作**：1-3 个核心动作（启用 / 停用 / 部署 / 同步 / 重启 / 删除等），按对象类型差异化

#### 字段类型 → 编辑方式映射

| 字段 | 编辑方式 | 组件 |
| --- | --- | --- |
| 名称 | hover 显示编辑入口，点击 inline 编辑 | `InputEditable` / `Editable` |
| 说明 / 描述 / 备注 | 同上 | `InputEditable` / `Editable` / `FormItemEditable` |
| 标签 | 就近 + 添加 / × 删除 | `ChipGroup` / `AppTag` |
| 状态 | 通过快捷动作按钮切换，附必要的二次确认 | `j-badge-status` + 动作按钮 |
| 归属 | 通常只读；如允许改，用单字段选择器 inline | `FormItemEditable` + 选择器 |
| 元信息 | 只读 | 文本 |

#### 用户行为细节（跨页面统一）

- hover 字段显示编辑入口（铅笔图标 / 下划线 / 浅色背景），让用户知道这里可改
- 点击字段直接进入 inline 编辑态
- Enter 保存、Esc 取消、失焦保存（具体按工作区统一约定）
- 校验失败就近气泡 / 红框提示，不跳出弹窗
- 保存成功用 toast 或区域提示，文案带对象名（例：`已更新设备 A 的名称`）
- 必填字段为空时回退到上一个有效值，并提示原因

#### 跨页面一致性

- 摘要区在 Tabs 切换时**始终保持可见**，不能随 Tab 内容滚走或重复展示
- 多个详情页（设备 / 产品 / 模型 / 通知配置 / AI 应用等）的头部摘要区结构必须复用同一套组件与节奏，避免每个详情页自造一套排版
- 快捷动作按钮的位置（右上角）、数量（1-3 + 更多）、命名风格在不同对象间保持一致

#### 反模式（明确禁止）

- ❌ 在头部摘要区右上角放一个"编辑"按钮，点击弹出包含名称 / 说明 / 标签等基本信息的编辑表单
- ❌ 头部只展示纯文本"名称：xxx"，没有 hover 显示的编辑入口
- ❌ 标签的增删必须进入编辑弹窗才能完成
- ❌ 把状态作为表单字段提供"切换状态"下拉，而不是用快捷动作
- ❌ 把基本信息（名称 / 说明 / 标签 / 状态）和复杂配置混在一个编辑表单里一起改
- ❌ 不同详情页的头部摘要区结构、字段顺序、动作位置都不一样
- ❌ 头部摘要区跟随 Tab 切换消失，或者每个 Tab 内重复一遍摘要

#### ASCII 骨架（统一参考）

```
┌────────────────────────────────────────────────────────────────────────────┐
│ [图标] 对象名（inline）           ● 状态徽标   [启用][停用][更多 ▾]       │
│ 说明（inline，长文截断 + tooltip / 展开）                                 │
│ 标签：[标签 1 ×] [标签 2 ×] [+ 添加]   归属：项目 A · 责任人 张三 · 区域 X│
│ 创建 2024-01-01 · 更新 2026-05-13 · 版本 v1.2                              │
└────────────────────────────────────────────────────────────────────────────┘
```

## 业务任务 → 方案路由

进入页面后用户的第一任务决定方案优先级。先做业务任务归类，再从对应方案组里挑 2-4 个候选给用户。

| 第一任务 | 推荐候选方案（按优先级） | 说明 |
| --- | --- | --- |
| 浏览定位一批对象 | 资产卡片台账页 / 筛选工作台页 / 标准管理表格页 | 看对象状态识别度与字段密度选择 |
| 横向对比多个对象 | 多对象对比页 / 资产卡片台账页 | 多于 2 个对象需要同视图比较时优先对比页 |
| 处置状态事件 | 处置工作台 / 筛选工作台页 / 时间线日志流 | 告警、工单、审批、巡检 |
| 持续维护单个对象 | 对象详情工作区 / 配置态-运行态切换页 / 主从详情工作区 | 看对象数量与运行态/编辑态分离需求 |
| 配置开通新能力 | 配置向导 / 资源选择器 | 分阶段配置或选资源后再开通 |
| 选择资源 / 模型 / 模板 | 资源选择器 / 配置向导 | 需要预览、筛选、对比时优先选择器 |
| 监控整体态势 | 监控感知页 / 分析钻取页 | 看是看"现在怎样"还是"为什么这样" |
| 分析指标与维度 | 分析钻取页 / 监控感知页 | 多维下钻、对比分析 |
| 查看事件演进 / 操作历史 | 时间线日志流页 / 主从详情工作区 | 需要按时序理解状态变化 |

> 同一业务任务在不同对象规模、状态复杂度、字段密度下会落到不同方案，给用户的 2-4 个候选必须说明业务差异，不是只换名字。

## 方案统一字段

每个方案按下面字段输出，缺哪项都会在落地时漂移：

```
- 业务任务
- 适用场景
- 不适用
- 首屏结构（ASCII 线框骨架）
- 信息密度目标
- 编辑触发梯度
- 核心组件（来自当前 workspace 的 jetlinks-web-core）
- 状态 / 标签锚点
- 主要动作位置
- 不借鉴清单
- 与相近方案的分界
```

## 方案目录

### 1. 标准管理表格页

#### 业务任务

批量检索、查看记录、做轻量增删改查；用户主要面对字段稳定的同类对象。

#### 适用场景

- 字段稳定、列结构清晰，对比依赖列
- 主要任务是按条件检索 + 批量操作
- 业务流程简单，对象状态不复杂

#### 不适用

- 表格字段过多、真正重点是先定位再看摘要
- 需要持续筛选、状态处置、对象工作区
- 业务核心价值不在表格本身

#### 首屏结构

```
┌─────────────────────── 标题 / 一句业务说明 ─────────────────────────┐
│ ConditionFilter（表达式输入 + Token） 或 ProSearch（旧页轻量筛选）  │
├──────────────────────────────────────────────────────────────────────┤
│ 表格工具栏：批量动作 · 新增 · 导入 / 导出 · 视图切换                │
├──────────────────────────────────────────────────────────────────────┤
│ j-pro-table                                                          │
│ ┌───┬─────┬─────┬─────┬─────┬─────┬─────┬─────┐                     │
│ │选 │字段1│字段2│状态 │字段4│字段5│字段6│操作 │                     │
│ ├───┼─────┼─────┼─────┼─────┼─────┼─────┼─────┤                     │
│ │   │ ... │ ... │ ●   │ ... │ ... │ ... │编辑 │                     │
│ └───┴─────┴─────┴─────┴─────┴─────┴─────┴─────┘                     │
│ 常规分页                                                             │
└──────────────────────────────────────────────────────────────────────┘
```

#### 信息密度目标

列数 6-10；操作列 1-2 个常用动作 + "更多"；行高紧凑；状态走 `j-badge-status`。

#### 编辑触发梯度

- 名称 / 备注 / 说明：行内 inline 编辑
- 完整记录：`EditDialog` 或 `JlDrawerShell`
- 复杂依赖：独立配置页

#### 核心组件

- `ConditionFilter`（默认）或 `ProSearch`（旧页 / 轻量固定筛选）
- `j-pro-table`
- `EditDialog` / `JlDrawerShell`
- `InputEditable` / `Editable` / `FormItemEditable`（轻量字段）
- `j-badge-status` / `MetaChip` / `AppTag`

#### 状态 / 标签锚点

状态列优先；分类、归属用胶囊；不靠纯文字表达状态。

#### 主要动作位置

工具栏（批量）+ 行内操作列（单条）。

#### 不借鉴清单

- 不要把"经营分析"的 4 KPI 卡套到管理表格页顶部
- 不要把告警处置台的"待处理 / 已处理"分栏挪进来
- 不要在表格上方再叠装饰 banner 与营销文案

#### 与相近方案的分界

- 字段少而对象状态强 → `资产卡片台账页`
- 高频组合筛选 + 状态处置 → `筛选工作台页` / `处置工作台`
- 第一任务是看单对象 → `对象详情工作区`

### 2. 资产卡片台账页

#### 业务任务

快速识别对象状态、标签、能力归属，并进入详情或就近动作。

#### 适用场景

- 设备资产、视频资源、模型、应用、通道、产品等对象
- 用户更依赖"状态 + 标签 + 摘要"快速判断，而不是字段平铺
- 卡片上有真实可识别的状态、能力、归属信息

#### 不适用

- 字段很少且用户需要密集列对比
- 对象没有可识别状态、标签或摘要

#### 首屏结构

```
┌─────────────────────── 标题 / 一句业务说明 ────────────────────────┐
│ ConditionFilter（表达式输入 + Token）                              │
├──────┬─────────────────────────────────────────────────────────────┤
│ 快捷 │ ┌── EntityCard ──┐ ┌── EntityCard ──┐ ┌── EntityCard ──┐  │
│ 筛选 │ │ 名称  状态徽标 │ │ 名称  状态徽标 │ │ 名称  状态徽标 │  │
│ 侧栏 │ │ 标签 标签 标签 │ │ 标签 标签 标签 │ │ 标签 标签 标签 │  │
│ (可  │ │ 摘要字段 1-3   │ │ 摘要字段 1-3   │ │ 摘要字段 1-3   │  │
│  选) │ │ [主动作][跳转] │ │ [主动作][跳转] │ │ [主动作][跳转] │  │
│      │ └────────────────┘ └────────────────┘ └────────────────┘  │
│      │ 轻量分页（总数 + 上一页 / 下一页）                          │
└──────┴─────────────────────────────────────────────────────────────┘
```

#### 信息密度目标

每行 3-4 张卡片；单卡 ≤ 5 个核心字段 + 状态徽标；标签 ≤ 3。

#### 编辑触发梯度

- 名称 / 标签 / 说明：进入详情后 inline 编辑
- 卡片不承担完整编辑，只承担识别 + 主动作 + 跳转

#### 核心组件

- `ConditionFilter`
- `QuickFilterSidebar`（可选）
- `ResponsiveGrid`
- `EntityCard` / `CardBox`
- `MetaChip` / `ChipGroup` / `AppTag` / `j-badge-status`
- `AIcon` / `IconLibrary`

#### 状态 / 标签锚点

卡片右上角放状态徽标；标签靠左排成胶囊；归属 / 类型用图标 + 文字表达。

#### 主要动作位置

卡片底部 1-2 个主动作 + 跳转入口；批量动作在顶部工具栏。

#### 不借鉴清单

- 不要在卡片上塞完整字段表
- 不要把卡片做成纯装饰容器
- 不要把卡片承担"完整编辑表单"

#### 与相近方案的分界

- 状态处置链路明显、需要左侧分组 → `筛选工作台页` / `处置工作台`
- 卡片只是入口、详情才是主战场 → `主从详情工作区` / `对象详情工作区`
- 用户更需要列对比 → `标准管理表格页`

### 3. 筛选工作台页

#### 业务任务

高频组合条件筛选 + 状态识别，定位到目标记录后就近处理。

#### 适用场景

- 告警记录、工单记录、订单台账、巡检记录、账号资产
- 记录字段较复杂，列表项需要表达业务重点
- 筛选比表格更重要，常用快捷筛选 / 已保存搜索 / 表达式筛选组合

#### 不适用

- 字段很少、普通表格已足够
- 主要目标是单对象编辑或配置
- 没有持续筛选需求

#### 首屏结构

```
┌─────────────────────── 标题 / 一句业务说明 ─────────────────────────┐
│ ConditionFilter（表达式输入主入口） + 已保存搜索                    │
├──────┬───────────────────────────────────────────────────────────────┤
│ 状态 │ 摘要列表 / 紧凑列表                                          │
│ 区域 │ ┌──────────────────────────────────────────────────────────┐ │
│ 已保 │ │ ● 状态徽标 · 名称 · 标签 ·  归属 · 时间 · [处理][查看]   │ │
│ 存搜 │ ├──────────────────────────────────────────────────────────┤ │
│ 索   │ │ ● 状态徽标 · 名称 · 标签 ·  归属 · 时间 · [处理][查看]   │ │
│ 自定 │ ├──────────────────────────────────────────────────────────┤ │
│ 义筛 │ │ ● 状态徽标 · 名称 · 标签 ·  归属 · 时间 · [处理][查看]   │ │
│ 选   │ └──────────────────────────────────────────────────────────┘ │
│      │ 轻量分页（总数 + 上一页 / 下一页）                            │
└──────┴───────────────────────────────────────────────────────────────┘
```

#### 信息密度目标

侧栏快捷筛选 ≤ 1 屏；列表单行 ≤ 7 个关键信息块；动作列只放 1-2 个核心动作。

#### 编辑触发梯度

- 状态切换 / 处理 / 指派：行内动作 + 局部抽屉
- 详情查看 / 完整处理：`JlDrawerShell` 或独立详情页
- 名称 / 备注：inline 编辑

#### 核心组件

- `ConditionFilter`
- `QuickFilterSidebar`
- 摘要列表 / `j-pro-table` 紧凑模式 / 卡片列表
- `JlDrawerShell`
- `j-badge-status` / `MetaChip` / `AppTag`

#### 状态 / 标签锚点

左侧侧栏聚焦"状态 / 区域 / 已保存搜索"，主筛选与侧栏共享同一条件模型，不分裂。

#### 主要动作位置

行内动作 + 顶部工具栏（批量）+ 抽屉内细化动作。

#### 不借鉴清单

- 不要把侧栏做成"导航树"风格的菜单
- 不要把筛选条件复制成两套独立模型
- 不要堆 KPI 卡占据筛选区上方空间

#### 与相近方案的分界

- 处置流程强、需要"待处理 / 处理中 / 已完成"连续动作 → `处置工作台`
- 卡片承载状态识别足够 → `资产卡片台账页`
- 用户最终聚焦单条详情持续编辑 → `主从详情工作区`

---

### 4. 多对象对比页

#### 业务任务

把 2-N 个对象放在同一视图中横向比较字段、状态、配置或趋势，辅助选择或审计。

#### 适用场景

- 设备 / 产品 / 模型 / 模板 / 配置 / 报表 的横向对比
- 选型场景、审计场景、变更前后对比
- 用户先选对象再看差异，差异比单值更重要

#### 不适用

- 只看一个对象 → `对象详情工作区`
- 对象很多、用户只是浏览 → `资产卡片台账页` / `筛选工作台页`
- 差异不重要、用户只看绝对值 → 普通详情或表格

#### 首屏结构

```
┌─────────────────────── 标题 / 一句业务说明 ────────────────────────┐
│ 对象选择条：[对象A ×] [对象B ×] [+ 添加对象] · 同步操作 · 导出     │
├──────────────────────────────────────────────────────────────────┤
│           │ 对象 A           │ 对象 B           │ 对象 C          │
│ 摘要      │ 状态 标签 归属   │ 状态 标签 归属   │ 状态 标签 归属  │
│           │ [跳详情]         │ [跳详情]         │ [跳详情]        │
├───────────┼──────────────────┼──────────────────┼─────────────────┤
│ 分区1     │ 字段值（差异高亮）│ 字段值           │ 字段值          │
│ 分区2     │ ...              │ ...              │ ...             │
│ 趋势 (可选)│ 折线 / 柱状      │ 折线 / 柱状      │ 折线 / 柱状     │
└───────────┴──────────────────┴──────────────────┴─────────────────┘
```

#### 信息密度目标

同时对比对象数 2-4，超过 4 用横向滚动；分区内字段 ≤ 8；差异行用前景色或徽标高亮。

#### 编辑触发梯度

- 对比页本身只读为主；如允许就地修改，单字段走 inline 编辑
- 复杂修改：跳转到对应对象的详情工作区

#### 核心组件

- `ConditionFilter` 或资源选择器（添加对比对象）
- `SectionCard` / `KvGrid`（按字段分区对比）
- `JEcharts`（趋势对比，必须有真实数据源）
- `MetaChip` / `j-badge-status`
- `StickyActionBar`（导出 / 同步操作）

#### 状态 / 标签锚点

对象摘要列首行表达状态 / 标签 / 归属；差异行用高亮区分；不要把所有字段都做强对比色，避免噪音。

#### 主要动作位置

顶部对象选择条 + 每列对象摘要"跳详情"+ 底部 / 顶部"导出对比报告"。

#### 不借鉴清单

- 不要照搬"经营分析对比大盘"的 KPI 矩阵
- 不要把对比页做成"两个独立详情页并排"
- 不要让用户在对比页里完成完整对象配置

#### 与相近方案的分界

- 只对比单对象不同时间点 → `时间线日志流页` 或 `分析钻取页`
- 对比维度其实是指标维度 → `分析钻取页`
- 选型场景同时需要预览能力描述 → `资源选择器`

### 5. 处置工作台

#### 业务任务

按优先级 / 时序识别问题，进入连续处置动作流，关键动作要少步骤、可追溯。

#### 适用场景

- 告警处置、工单流转、审批中心、巡检任务、运维事件
- 用户进入页面是为了"先识别 → 再决策 → 再执行 → 留痕"
- 状态流程明显（待处理 / 处理中 / 已完成）

#### 不适用

- 没有连续动作、只是看一批记录 → `筛选工作台页`
- 主要任务是配置或编辑单对象 → `对象详情工作区` / `配置向导`

#### 首屏结构

```
┌─────────────────────── 标题 / 一句业务说明 ──────────────────────────┐
│ 状态分栏：[待处理 12] [处理中 5] [已完成 ...]  · 我的待办 · 全部     │
├───────────────────────────────────────┬──────────────────────────────┤
│ 任务列表（默认按优先级 / 时间排序）   │ 当前任务详情面板（右侧）     │
│ ┌────────────────────────────────┐    │ ┌────────────────────────┐  │
│ │ 高 · 名称 · 标签 · 时间 · [处理]│    │ │ 对象摘要 / 状态 / 归属│  │
│ ├────────────────────────────────┤    │ ├────────────────────────┤  │
│ │ 中 · 名称 · 标签 · 时间 · [处理]│    │ │ 处置历史 / 时间线      │  │
│ ├────────────────────────────────┤    │ ├────────────────────────┤  │
│ │ 低 · 名称 · 标签 · 时间 · [处理]│    │ │ 处置表单 / 快捷动作    │  │
│ └────────────────────────────────┘    │ │ [处理][指派][转单]     │  │
│                                       │ └────────────────────────┘  │
└───────────────────────────────────────┴──────────────────────────────┘
```

#### 信息密度目标

左侧任务行 ≤ 6 个关键信息块；右侧详情按"摘要 / 历史 / 当前动作"三段；不堆叠 KPI 卡。

#### 编辑触发梯度

- 状态切换 / 指派 / 转单：右侧详情面板内的快捷动作
- 备注 / 处置说明：行内 inline 或 `FormItemEditable`
- 完整跳转：进入对象详情工作区

#### 核心组件

- 状态分栏（可用 `MetaChip` / `ChipGroup` 加计数）
- `QuickFilterSidebar`（按优先级 / 责任人 / 区域过滤）
- 摘要列表 / `j-pro-table` 紧凑模式
- `SectionCard` + `KvGrid` + Timeline（处置历史）
- `StickyActionBar`（底部主动作）

#### 状态 / 标签锚点

状态分栏 + 行优先级徽标；处置历史用时间线表达"先后次序 + 处置人 + 关键变化"。

#### 主要动作位置

右侧详情底部固定动作栏 + 行内快捷处置 + 顶部批量处理（少用）。

#### 不借鉴清单

- 不要把处置台压成"普通表格 + 编辑弹窗"
- 不要把状态分栏做成静态文字而不能切换数据集
- 不要在右侧详情塞与处置无关的概览图

#### 与相近方案的分界

- 没有连续动作 → `筛选工作台页`
- 单对象长期维护 → `对象详情工作区`
- 看历史演进而非当前处置 → `时间线日志流页`

### 6. 对象详情工作区

#### 业务任务

围绕单个对象持续查看、配置、排查、维护；用户在这里待最久。

#### 适用场景

- 设备 / 产品 / 模型 / 通知配置 / AI 应用 / 规则 等单对象
- 名称、标签、说明、状态、归属需要局部编辑
- 关联记录、日志、配置、运行状态都集中在一处

#### 不适用

- 用户第一任务是批量筛选或处理
- 详情只有少量字段且不需要持续维护

#### 首屏结构

```
┌─────────────────────── 对象摘要区 ────────────────────────────────────┐
│ [图标] 对象名（inline 可改）   状态徽标 · 标签 · 归属 · 创建/更新时间│
│ 快捷动作：[启用/停用] [部署] [更多]                                  │
├──────────────────────────────────────────────────────────────────────┤
│ Tabs：基本信息 · 运行状态 · 配置 · 关联记录 · 日志                   │
├──────────────────────────────────────────────────────────────────────┤
│ ┌─ SectionCard：基本信息 ─────────┐ ┌─ SectionCard：归属与权限 ───┐ │
│ │ KvGrid（名称 / 描述 inline 可改）│ │ KvGrid                       │ │
│ └────────────────────────────────┘ └─────────────────────────────┘ │
│ ┌─ SectionCard：能力 / 配置 ─────────────────────────────────────┐ │
│ │ 子配置入口 / 状态切换 / 局部编辑                                 │ │
│ └────────────────────────────────────────────────────────────────┘ │
├──────────────────────────────────────────────────────────────────────┤
│ StickyActionBar（保存 / 取消 / 全局动作）                           │
└──────────────────────────────────────────────────────────────────────┘
```

#### 信息密度目标

单 Tab 内分区 2-4 个；分区内字段 ≤ 8；摘要区只表达"是什么 + 怎么样 + 谁的"。

#### 头部摘要区规范

按"反传统后台感硬约束 - 7. 详情页头部摘要区"实现：名称 / 说明 / 描述 / 备注 inline 可改；标签就近 + / × 增删；状态用徽标 + 快捷动作切换；归属与元信息只读；禁止"编辑按钮 → 弹出基本信息表单"。

#### 编辑触发梯度

- 名称 / 描述 / 备注 / 标签 / 状态：摘要区或 `KvGrid` 中按头部摘要区规范处理（inline 或就近动作）
- 单分区配置：`SectionCard` 内部分区编辑态 + `StickyActionBar`
- 跨分区强校验：`JlDrawerShell` 或独立子配置页

#### 核心组件

- `SectionCard`
- `KvGrid`
- `TabsCard`
- `InputEditable` / `Editable` / `FormItemEditable`
- `StickyActionBar`
- `j-badge-status` / `MetaChip` / `AppTag`

#### 状态 / 标签锚点

摘要区第一行表达"对象身份 + 状态 + 标签 + 归属"，避免把这些信息散落到下方 Tab。

#### 主要动作位置

摘要区右上角放快捷动作；分区内放局部动作；`StickyActionBar` 放全局保存。

#### 不借鉴清单

- 不要把详情页做成"一个超长表单 + 底部保存"
- 不要把首屏堆满与当前对象无关的统计卡
- 不要在每个分区都重复对象摘要

#### 与相近方案的分界

- 同时浏览一批对象 + 看单详情 → `主从详情工作区`
- 运行态查看 vs 编辑态修改要明确分离 → `配置态-运行态切换页`
- 强阶段配置 → `配置向导`

### 7. 主从详情工作区

#### 业务任务

左侧浏览一批对象 / 节点，右侧聚焦单对象详情、配置、关联记录，频繁切换上下文。

#### 适用场景

- 设备分组、项目成员、模板配置、规则节点、菜单管理、产品物模型
- 列表本身不复杂，但单对象详情很重，需要并行操作

#### 不适用

- 第一任务是批量筛选记录 → `筛选工作台页`
- 单对象本身详情很轻 → `标准管理表格页` + `EditDialog`

#### 首屏结构

```
┌─────────────────────── 标题 / 一句业务说明 ──────────────────────────┐
│ ┌── 左侧导航 ──────┐ │ ┌── 右侧详情工作区 ─────────────────────────┐ │
│ │ 搜索 / 过滤      │ │ │ 对象摘要：名称 状态 标签 快捷动作         │ │
│ │ ▾ 分组 1         │ │ ├──────────────────────────────────────────┤ │
│ │   • 节点 A (选中)│ │ │ Tabs：详情 · 关联 · 日志                  │ │
│ │   • 节点 B       │ │ ├──────────────────────────────────────────┤ │
│ │ ▾ 分组 2         │ │ │ SectionCard / KvGrid / inline 编辑       │ │
│ │   • 节点 C       │ │ │ ...                                       │ │
│ │ ...              │ │ │ StickyActionBar                           │ │
│ └──────────────────┘ │ └──────────────────────────────────────────┘ │
└─────────────────────────────────────────────────────────────────────┘
```

#### 信息密度目标

左侧导航单行只表达节点名 + 状态徽标；右侧详情按对象详情工作区的密度执行。

#### 头部摘要区规范

右侧"对象摘要"按"反传统后台感硬约束 - 7. 详情页头部摘要区"实现：名称 / 说明 inline 可改；标签就近 + / × 增删；状态用徽标 + 快捷动作切换；禁止"编辑按钮 → 基本信息表单"。

#### 编辑触发梯度

- 节点名 / 节点描述：左侧 inline 或右侧摘要 inline（按头部摘要区规范）
- 分区配置：右侧 `SectionCard` 内编辑态 + `StickyActionBar`
- 节点新增 / 删除：左侧右键 / 工具栏

#### 核心组件

- 左侧：树 / 列表 / 分组列表组件（按工作区现状选）
- 右侧：`SectionCard` / `KvGrid` / `TabsCard` / `InputEditable`
- `StickyActionBar`
- `j-badge-status`

#### 状态 / 标签锚点

左侧节点表达状态徽标；选中节点用强对比；分组用 `ChipGroup` 计数（可选）。

#### 主要动作位置

左侧工具栏 / 右键：节点级动作；右侧顶部：对象快捷动作；右侧底部：保存。

#### 不借鉴清单

- 不要把左侧节点做成"超宽多列表格"
- 不要让左侧承担字段对比，主对比应该走 `多对象对比页`
- 不要让左右两侧重复展示同一份摘要信息

#### 与相近方案的分界

- 单对象单任务 → `对象详情工作区`
- 用户更需要状态分栏与连续处置 → `处置工作台`
- 节点之间需要并排对比 → `多对象对比页`

### 8. 配置态-运行态切换页

#### 业务任务

让用户在"看现在怎么跑"和"改它怎么跑"之间明确切换，避免运行态被编辑态污染。

#### 适用场景

- 设备配置 / 规则引擎 / 数据源 / 协议适配 / 通知通道 等
- 运行态和编辑态信息差异显著（运行态是状态 + 指标，编辑态是字段 + 校验）
- 误编辑成本高，需要二次确认

#### 不适用

- 编辑态本身就是运行态（如标签、备注）
- 没有运行态信息 → `对象详情工作区`

#### 首屏结构

```
┌─────────────────────── 对象摘要 + 模式切换 ───────────────────────────┐
│ [图标] 对象名 · 状态徽标 · 归属                                     │
│ 模式切换：(● 运行态) (○ 编辑态)   · [更多]                          │
├──────────────────────────────────────────────────────────────────────┤
│ 运行态视图（默认）                                                   │
│ ┌── 实时状态 ─────┐ ┌── 关键指标 ─────┐ ┌── 最近事件 ──────┐       │
│ │ 运行 / 在线 / 离 │ │ 数值 + 阈值     │ │ 时间线（精简）   │       │
│ └────────────────┘ └────────────────┘ └─────────────────┘       │
│                                                                      │
│ 切换到编辑态后：                                                     │
│ ┌── SectionCard：配置分区 1 ────────────────────────────────────┐   │
│ │ KvGrid / FormItemEditable / 分区局部编辑                        │   │
│ └────────────────────────────────────────────────────────────┘   │
│ StickyActionBar（保存 / 取消 / 二次确认）                            │
└──────────────────────────────────────────────────────────────────────┘
```

#### 信息密度目标

运行态：状态 + 指标 + 最近事件，三段 ≤ 1 屏；编辑态：分区表单 + 校验。

#### 头部摘要区规范

按"反传统后台感硬约束 - 7. 详情页头部摘要区"实现：名称 / 说明 inline 可改；标签就近 + / × 增删；状态徽标 + 快捷动作切换。模式切换控件本身放在摘要区右侧或紧邻位置，禁止"编辑按钮 → 基本信息表单"。

#### 编辑触发梯度

- 进入编辑态由顶部模式切换控制，不允许在运行态直接修改强校验字段
- 名称 / 说明 / 备注 / 标签等轻量字段两态都按头部摘要区规范 inline 编辑
- 关键配置变更：编辑态 + 二次确认弹窗

#### 核心组件

- 模式切换：Tabs 或 Segmented（按工作区组件可用性）
- 运行态：`SectionCard` + `KvGrid` + `JEcharts` + Timeline
- 编辑态：`SectionCard` + `FormItemEditable` + 校验
- `StickyActionBar`

#### 状态 / 标签锚点

模式切换本身用强对比色 / 选中态；运行态以状态徽标和指标为主；编辑态以字段标签和必填提示为主。

#### 主要动作位置

模式切换在摘要区右侧；编辑态保存在 `StickyActionBar`；运行态主动作（重启 / 同步 / 部署）在摘要区右上角。

#### 不借鉴清单

- 不要把"运行 + 编辑"挤在同一个表单视图里
- 不要把模式切换做成隐藏入口或多层菜单
- 不要在运行态视图里暴露未保存的编辑草稿

#### 与相近方案的分界

- 没有运行态信息差 → `对象详情工作区`
- 编辑流程强阶段 → `配置向导`
- 需要左侧浏览节点 → `主从详情工作区`

### 9. 配置向导 / 能力开通页

#### 业务任务

按阶段完成一次配置 / 录入 / 开通，每一步只暴露当前阶段必须的信息。

#### 适用场景

- 能力开通、通道接入、协议接入、首次配置、产品发布
- 阶段顺序与校验依赖明显
- 用户主要操作一个对象而不是一批对象

#### 不适用

- 阶段不明显的普通表单 → `EditDialog` 或对象详情工作区
- 需要持续切换多条对象 → `主从详情工作区`

#### 首屏结构

```
┌─────────────────────── 标题 / 一句业务说明 ─────────────────────────┐
│ Steps：(1 选能力) → (2 配置参数) → (3 校验联通) → (4 完成)        │
├──────────────────────────────────────────────────────────────────────┤
│ 当前阶段（示例：2 配置参数）                                        │
│ ┌── 能力卡片（已选）：名称 + 摘要 + 切换链接 ────────────────────┐  │
│ └──────────────────────────────────────────────────────────────┘  │
│ ┌── SectionCard：基本参数 ─────────────────────────────────────┐  │
│ │ FormItem ...                                                    │  │
│ └──────────────────────────────────────────────────────────────┘  │
│ ┌── SectionCard：高级参数（可折叠） ────────────────────────────┐  │
│ │ FormItem ...                                                    │  │
│ └──────────────────────────────────────────────────────────────┘  │
├──────────────────────────────────────────────────────────────────────┤
│ StickyActionBar：[上一步] [保存草稿] [下一步]                       │
└──────────────────────────────────────────────────────────────────────┘
```

#### 信息密度目标

单阶段表单 ≤ 12 字段；超出就拆"基本 / 高级 / 联通验证"分区；每步都给出"完成此步意味着什么"。

#### 编辑触发梯度

- 各阶段都是表单编辑态本身，不存在"先看后改"
- 草稿态允许 inline 修改 + 校验

#### 核心组件

- Steps / 分段导航
- `CardSelect`（能力 / 通道 / 模板选择）
- `SectionCard` + 表单
- `StickyActionBar`

#### 状态 / 标签锚点

Steps 当前态 / 完成态 / 待办态强对比；能力卡片选中态明确。

#### 主要动作位置

`StickyActionBar` 固定底部："上一步 / 保存草稿 / 下一步 / 完成"。

#### 不借鉴清单

- 不要把强阶段流压成单页超长表单
- 不要把"完成阶段动作"放在页面右上角离主流程远的位置
- 不要在向导内插入与本次开通无关的统计卡

#### 与相近方案的分界

- 选资源后跳到详情工作区做长期维护 → 进入 `对象详情工作区`
- 没有阶段依赖 → `EditDialog` 或对象详情局部编辑
- 选资源本身需要预览 / 对比 → `资源选择器`

### 10. 资源 / 能力选择器

#### 业务任务

在多个能力 / 模型 / 模板 / 插件 / 市场资源中筛选、预览、对比，再确认选择。

#### 适用场景

- 协议选择、模型选择、模板选择、插件市场、第三方能力接入
- 选项较多，需要筛选 + 预览 + 能力说明
- 用户在选项之间犹豫，需要支持比较与试用

#### 不适用

- 选项极少且没有预览价值 → 普通下拉
- 选完即用，不需要细看 → `CardSelect` 单步即可

#### 首屏结构

```
┌─────────────────────── 标题 / 一句业务说明 ─────────────────────────┐
│ ConditionFilter / 标签筛选 / 分类切换                                │
├─────────────────────────────────────┬──────────────────────────────┤
│ 卡片网格（资源 / 能力卡片）         │ 右侧预览区（点选后展开）       │
│ ┌─ ResourceCard ─┐ ┌─ ResourceCard ┐│ ┌── 能力说明 ────────────┐  │
│ │ 图标 名称 类别 │ │ 图标 名称 类别 ││ │ 简介 / 用法 / 限制      │  │
│ │ 摘要 标签      │ │ 摘要 标签     ││ │ 参数预览 / 截图         │  │
│ │ [预览] [选择]  │ │ [预览] [选择] ││ │ [选择] [对比]           │  │
│ └────────────────┘ └───────────────┘│ └────────────────────────┘  │
└─────────────────────────────────────┴──────────────────────────────┘
```

#### 信息密度目标

每行 3-4 张资源卡；预览区按"简介 / 参数 / 截图 / 限制"分块；可选对比 ≤ 4 个。

#### 编辑触发梯度

- 选择器本身不承担编辑，选完后由配置向导或对象详情工作区承接
- 草稿态可允许"对比"" 收藏""加入选择栏"

#### 核心组件

- `MarketplaceResourcePicker`
- `CardSelect`
- `ConditionFilter`
- `ChipGroup` / `MetaChip`
- 预览区：`SectionCard` + `KvGrid`

#### 状态 / 标签锚点

资源卡用标签 / 分类 / 能力等级表达；预览区状态徽标表达"可用 / Beta / 不可用"。

#### 主要动作位置

资源卡内："预览 / 选择"；预览区右下角："选择 / 对比"；底部"已选清单"（如允许多选）。

#### 不借鉴清单

- 不要把选择器退化为下拉 + 表格
- 不要把市场资源做成纯营销 banner
- 不要在选择阶段强行进入完整配置表单

#### 与相近方案的分界

- 选完后要做长期配置 → 进入 `配置向导` / `对象详情工作区`
- 需要并排细看差异 → `多对象对比页`
- 资源数量极少 → 普通下拉或 `CardSelect`

### 11. 监控感知页

#### 业务任务

让用户在几秒内理解整体运行态势、关键状态分布与异常聚焦，并能就近 drill-down。

#### 适用场景

- 设备监控、平台运行态、规则运行态、消息总线、协议运行态
- 用户主要看"现在怎样 + 哪里有问题"
- 每个指标 / 图表都能回答"它驱动什么决策、来自哪里、下一步去哪"

#### 不适用

- 没有真实数据源或刷新机制
- 用户的真实任务是 CRUD 或处置
- 只是想让页面"看起来丰富"

#### 首屏结构

```
┌─────────────────────── 标题 / 一句业务说明 ─────────────────────────┐
│ 时间范围 / 范围筛选 / 刷新频率                                       │
├──────────────────────────────────────────────────────────────────────┤
│ ┌── 关键状态分布 ─────┐ ┌── 异常聚焦区 ──────────────────────────┐ │
│ │ 状态环 / 状态分栏    │ │ ● 高优异常 · 名称 · 影响范围 · [处置] │ │
│ │ + 计数 + 占比        │ │ ● 中优异常 · ...                       │ │
│ └────────────────────┘ │ ● 趋势异常 · ...                       │ │
│                        └────────────────────────────────────────┘ │
├──────────────────────────────────────────────────────────────────────┤
│ ┌── 趋势 / 分布图 (JEcharts) ─────────────────────────────────────┐ │
│ │ 真实指标 + 真实时间窗口 + 下钻入口                                │ │
│ └────────────────────────────────────────────────────────────────┘ │
├──────────────────────────────────────────────────────────────────────┤
│ Drill-down 区：异常详情 / 受影响对象列表 / 跳处置台                 │
└──────────────────────────────────────────────────────────────────────┘
```

#### 信息密度目标

首屏 ≤ 3 个一级区块；每个区块只解决一个"判断或动作"问题；不堆装饰卡。

#### 编辑触发梯度

- 监控感知页通常只读；阈值 / 范围配置进入对象详情工作区或配置向导
- 异常区允许 inline "确认 / 抑制 / 转处置"

#### 核心组件

- `JDashboardTimeSelect`
- `JEcharts`
- 状态分栏 / `MetaChip`
- 异常聚焦：摘要列表
- Drill-down：跳转到处置台或对象详情

#### 状态 / 标签锚点

状态分布用强对比色环；异常区按优先级分高 / 中 / 低；趋势图用真实阈值与异常点高亮。

#### 主要动作位置

异常行内："处置 / 查看"；趋势图下钻按钮；时间范围切换在顶部。

#### 不借鉴清单

- 不要照搬"经营分析大盘"的 KPI 数字墙
- 不要在没有真实来源时引入占位图表
- 不要把感知页变成"运营海报"

#### 与相近方案的分界

- 用户重点是"为什么这样" → `分析钻取页`
- 用户重点是"现在该做什么" → `处置工作台`
- 用户重点是"事件演进" → `时间线日志流页`

### 12. 分析钻取页

#### 业务任务

从指标到维度再到对象 / 事件，逐层下钻，理解原因、对比维度、定位异常。

#### 适用场景

- 设备运行分析、消息吞吐分析、能力调用分析、告警归因
- 用户需要"按维度看指标 → 选维度下钻 → 看对象 / 事件"
- 指标和维度都有真实数据来源

#### 不适用

- 没有维度结构，只是单一时间序列 → `监控感知页`
- 用户真实任务是处置或配置

#### 首屏结构

```
┌─────────────────────── 标题 / 一句业务说明 ──────────────────────────┐
│ 时间范围 + 维度筛选（区域 / 类型 / 协议 / 责任人 ...）              │
├──────────────────────────────────────────────────────────────────────┤
│ ┌── 总览指标 ──┐ ┌── 维度分布 ─────────────────────────────────┐  │
│ │ 数值 + 同比  │ │ 柱状 / 堆叠 / 环：按当前维度切片              │  │
│ └─────────────┘ └────────────────────────────────────────────┘  │
├──────────────────────────────────────────────────────────────────────┤
│ ┌── 钻取面包屑 ────────────────────────────────────────────────┐   │
│ │ 全部 > 区域 A > 产品类型 B > 设备列表                         │   │
│ └──────────────────────────────────────────────────────────────┘   │
│ ┌── 当前层级数据：列表 / 卡片 / 表格 ─────────────────────────────┐ │
│ │ 行内 [钻取] 或点击进入下一层级                                   │ │
│ └────────────────────────────────────────────────────────────────┘ │
└──────────────────────────────────────────────────────────────────────┘
```

#### 信息密度目标

总览指标 ≤ 4 个；维度分布只画 1 张主图；钻取层级 ≤ 4 级；当前层级列表行紧凑。

#### 编辑触发梯度

- 钻取页只读；如允许标记 / 备注，走 inline
- 进一步处置：进入处置工作台或对象详情

#### 核心组件

- `JDashboardTimeSelect`
- `JEcharts`
- 钻取面包屑
- 摘要列表 / `j-pro-table` 紧凑模式
- `ConditionFilter`

#### 状态 / 标签锚点

按维度切片时用一致的色板；异常切片高亮；面包屑作为强导航锚点。

#### 主要动作位置

钻取交互在图表 + 列表内；顶部切换维度 + 时间范围；底部少量批量动作。

#### 不借鉴清单

- 不要把钻取页堆成"图表瀑布墙"
- 不要在没有维度数据时硬塞维度切换
- 不要让用户钻取后回不到上一层（面包屑必须可点）

#### 与相近方案的分界

- 用户更关心"现在态势" → `监控感知页`
- 钻到对象后要持续维护 → `对象详情工作区`
- 时序事件分析 → `时间线日志流页`

### 13. 时间线 / 日志流页

#### 业务任务

按时序理解事件演进、操作历史或状态变化，定位关键节点。

#### 适用场景

- 设备事件、操作日志、消息回放、规则触发链、审计日志、变更记录
- 用户决策依赖"先发生什么 → 后发生什么 → 为何变化"
- 节点之间有因果或顺序关系

#### 不适用

- 用户更需要状态分布 → `监控感知页`
- 没有时序结构，只是字段列表 → `标准管理表格页`

#### 首屏结构

```
┌─────────────────────── 标题 / 一句业务说明 ─────────────────────────┐
│ 时间范围 + 类型 / 级别 / 来源筛选                                    │
├──────┬───────────────────────────────────────────────────────────────┤
│ 左侧 │ 时间线主区                                                    │
│ 类型 │  ●─── 时间 · 类型 · 摘要 · [详情]                            │
│ /级别│  │                                                            │
│ 过滤 │  ●─── 时间 · 类型 · 摘要 · [详情]                            │
│      │  │                                                            │
│ 已保 │  ●─── 时间 · 类型 · 摘要 · [详情]                            │
│ 存视 │  │     展开节点 → 关键参数 / 关联对象 / 处置                  │
│ 图   │  ●─── 时间 · 类型 · 摘要 · [详情]                            │
└──────┴───────────────────────────────────────────────────────────────┘
```

#### 信息密度目标

时间线节点单行 ≤ 5 个关键信息块；展开后详情按"参数 / 关联 / 处置"三段；左侧过滤项 ≤ 1 屏。

#### 编辑触发梯度

- 通常只读；如允许标注 / 备注，走 inline
- 处置 / 转单：跳处置工作台

#### 核心组件

- 时间线组件（按工作区现状选；若无统一组件，用紧凑列表 + 节点圆点 + 连线 CSS）
- `QuickFilterSidebar`
- `ConditionFilter`
- `SectionCard`（节点展开详情）
- `MetaChip` / `j-badge-status`

#### 状态 / 标签锚点

节点类型 / 级别用图标 + 颜色双锚点；高亮关键变更（状态翻转、配置变更）。

#### 主要动作位置

节点行 / 展开区："详情 / 处置 / 关联"；顶部时间范围 + 类型筛选。

#### 不借鉴清单

- 不要把时间线压扁成"普通日志表格"
- 不要为了视觉而把所有节点画成一样的彩色圆点
- 不要在时间线上展示与时序无关的指标卡

#### 与相近方案的分界

- 用户需要按维度切片分析 → `分析钻取页`
- 用户重点是处置当前状态 → `处置工作台`
- 节点详情非常重 → `对象详情工作区` 或抽屉

## 方案档案输出

用户选定后，给后续实现保留一份简短方案档案：

- solutionName
- primaryFilterSurface
- contentSurface
- detailCarrier
- editMode
- coreComponents
- actionPlacement
- densityTarget
- sidebarMode

这份档案只锁定业务结构和交互路径，不代表组件已经可用；落地时仍必须回到 `jetlinks-web` 核验当前 workspace 的 `jetlinks-web-core` 导出、组件契约和相邻页面用法。

## 方案选择时的推荐措辞

推荐固定用下面格式：

1. 推荐：`资产卡片台账页`
   - 适合：快速识别对象状态、标签和主动作
   - 结构：顶部条件筛选 + 可选快捷筛选 + 卡片台账
   - 组件：`ConditionFilter` + `ResponsiveGrid` + `EntityCard` / `CardBox`
2. 备选：`筛选工作台页`
   - 适合：高频组合条件定位复杂记录
   - 结构：顶部条件筛选 + 左侧快捷筛选 + 摘要列表
3. 备选：`处置工作台`
   - 适合：状态分栏 + 连续动作流（告警 / 工单 / 巡检）
4. 备选：`对象详情工作区`
   - 适合：围绕单个对象持续查看、局部编辑和配置

如果用户没有明确偏好，给出默认项并直接说明原因。
