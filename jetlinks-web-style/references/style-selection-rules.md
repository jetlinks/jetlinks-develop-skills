# 页面风格选择规则

## 什么时候必须先选风格

命中任一项时，先做风格选择：

- 用户明确说“参考现在这个页面的风格”“沿用资产页这种结构”
- 页面既可能做成标准表格页，也可能做成工作台 / 台账 / 主从结构
- 用户要求“更现代”“更适合运营使用”“不要普通表格页”，但没有给出明确页面骨架
- 页面重构重点是布局、信息架构、交互组织，而不是单个组件细节

如果页面只是局部调整，例如改一个表单、补一个筛选项、替换一个弹窗，不要额外开启风格选择流程。

## 选择流程

### 1. 先做业务归类

先判断：

- 用户进入页面后的第一任务是什么
- 用户主要在看一批对象，还是深度处理一个对象
- 页面主要价值在“定位记录”“批量处理”“持续监控”“深入编辑”中的哪一种

### 2. 只给少量具体选项

给 2-4 个选项，且每个选项都要说明：

- 适合什么业务
- 结构上会长什么样
- 为什么推荐或不推荐

不要问：

- “你想要什么风格？”
- “你喜欢哪种设计？”
- “你能再描述一下吗？”

应改成：

- “这个页面我建议在以下 3 种结构里选一种，我默认按 A 继续。”

### 3. 默认项必须明确

如果用户没有额外偏好，给出一个默认项并说明理由。

推荐表达：

- “如果你没有特别偏好，我默认按 `筛选工作台式台账页` 继续，因为这个页面的第一任务是高频筛选后定位记录。”

### 4. 选定后固化为 style profile

一旦用户选定风格，后续实现都围绕这个 profile，不要在实现途中再次漂移。

最少记录这 6 项：

- styleName
- primaryFilterSurface
- contentSurface
- actionPlacement
- densityTarget
- sidebarMode

## 现有映射规则

### 用户说“按现在资产页这种风格”

直接映射为：

- `筛选工作台式台账页`

### 用户说“就普通后台表格那种”

直接映射为：

- `标准管理表格页`

### 用户说“左侧树 / 列表，右侧详情编辑”

直接映射为：

- `主从详情工作区`

## 扩展规则

后续新增风格时，不改主流程，只在 style catalog 中增加：

- 风格名
- 适用场景
- 页面骨架
- 不适用场景
- 与相近风格的分界
