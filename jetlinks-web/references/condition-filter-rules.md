# JetLinks 条件筛选与表达式搜索规则

本文件用于 JetLinks 前端中实现通用筛选组件、表达式化条件输入、条件路由回显、远程选项筛选以及快捷筛选联动时的稳定做法。

## 何时使用

- 需要实现一个输入框式、Token 化、连续交互的条件筛选
- 需要基于 `ConditionFilter` 或同类条件模型输出 `terms` / `where` / `q`
- 需要让业务方只配置字段、条件和值编辑器，而不重复造交互外壳
- 需要远程选项筛选、选中值反显、已保存搜索、快捷筛选联动

## 核心原则

1. 先复用统一条件模型，不自造第二套协议
   - 优先复用当前 workspace 已有的条件模型、路由编码和回显工具。
   - 如果已有 `ConditionFilter`、`encodeConditionFilterQuery`、`decodeConditionFilterQuery`，优先沿用。
   - 不要一边维护 `terms`，一边再额外维护一套业务私有筛选对象。

2. 组件只负责交互壳，不绑死业务值编辑逻辑
   - 通用筛选组件负责字段选择、条件切换、Token 展示、光标行为、失焦/回车生效和自动搜索节奏。
   - 字段条件集合和值输入方式应由字段定义驱动，而不是写死在组件内部。

3. 表达式输入优先保持“所见即所得”
   - 外层就是输入框，不再拆成“查看态 / 编辑态”。
   - 已生成条件应直接以内联 Token 展示，并可被光标、退格、点击再次编辑。
   - 不要要求用户进入单独编辑弹窗后再确认。

4. 同一个条件只做一件事
   - 条件类型由条件 Token 负责。
   - 值编辑器只负责值本身。
   - 如果已经进入值面板编辑，不要在值面板里再重复渲染一遍条件选择。

5. 路由回显必须与运行态完全同源
   - `q` 的编码、反解码、界面回显、接口请求体构造，必须共用同一组字段元数据与转换逻辑。
   - 不要让 URL、筛选框展示和实际请求三套逻辑各自演化。

## 推荐模式

### 一、字段定义驱动筛选能力

适用：
- 业务方传入字段、条件类型、值编辑器或远程选项源

要求：
- 字段定义至少包含：展示名、数据列、类型、可用条件、值编辑方式。
- 如需显示字段名但按别的列查询，优先使用字段级重命名能力（如 `rename`），而不是在外部额外拼接请求。
- 如需将界面条件映射成自定义后端条件，优先使用字段级转换入口（如 `handleParamsItem`）。

### 二、表达式式输入框

适用：
- GitHub / YouTrack 风格的连续条件编辑

要求：
- 输入字段名后，应直接给出建议项；完全匹配字段时优先视为“选择字段”，而不是“字段 + 纯文本值”。
- 条件 Token、值 Token 都应可点击重新编辑。
- Tab、Enter、Blur 的行为要稳定：
  - Enter：提交当前值或当前选择
  - Blur：视为生效，不要求额外点击“应用”
  - Tab：优先进入下一个编辑节点，不要误落到删除按钮

### 三、默认条件集合与条件文案

适用：
- 通用筛选组件需要对不同字段类型给出开箱即用的常用条件

要求：
- 默认条件集合应按字段类型分组，而不是全字段共用一套。
- 对字符串、数值、日期、选项字段分别提供合理默认值。
- 默认推荐条件应按语义而不是按技术类型决定，例如：
  - ID 类字段优先“等于”
  - 文本描述类优先“包含”
  - 日期类优先“范围”
  - 选项类优先“多选 in”
- 展示文案应以用户语义为主，例如“为”“不为”“处于范围”“包含”，不要直接暴露生硬 DSL。

### 四、无值条件与值清空策略

适用：
- `isnull` / `notnull` 这类不需要用户输入值的条件

要求：
- 该类条件切换后应立即生效，不再打开值编辑。
- 如果后端契约要求无值条件仍传固定值，占位值必须在字段转换层统一处理，不要散落在页面代码里。
- 当用户还没输入值就切换条件或字段时，不要粗暴删除整个条件；优先保留当前编辑上下文。

### 五、远程选项面板

适用：
- 选项来自接口、字典、用户、项目、树形结构、区域等

要求：
- 远程数据优先走面板式选择，而不是在输入框里堆一个难以复用的下拉。
- 接口过滤必须由 `loadOptions` 一类远程能力驱动，不能只在本地数组上过滤。
- 选中值反显必须依赖 `loadSelectedOptions` 一类按值回查能力，不能只显示原始 id。
- 对远程搜索加防抖，避免高频请求；首次打开与关键字变化时再查询，不要每次勾选都重新拉取。
- 多选场景默认按 `in` 组织，组件层只维护选中集，不擅自改写业务条件语义。

### 六、显示值与查询值分离

适用：
- 显示名称、实际提交 id；或显示业务字段、实际走关联条件查询

要求：
- Token 上展示用户能理解的名称。
- 请求中提交后端真正需要的列和值。
- 对路由回显场景，必须能仅凭已编码条件重新拉取显示标签。

### 七、快捷筛选与主筛选联动

适用：
- 左侧快捷筛选、状态快捷项、已保存搜索、顶部筛选框共存

要求：
- 快捷筛选应复用与主筛选完全相同的 `terms` 模型，不维护第二份真值。
- 左侧面板或快捷状态项只负责生成 / 替换标准条件，不直接耦合表格请求实现。
- 已保存搜索优先保存标准化后的 `q` 或统一条件模型，而不是单独保存业务私有结构。

### 八、路由参数设计

适用：
- 需要在 URL 中保存复杂筛选条件并支持回显

要求：
- 优先使用统一压缩编码与字段短别名（如 `routeAlias`）来缩短 `q`。
- 不要直接在 URL 中裸拼完整 `terms` JSON。
- 只有字段映射、条件映射、值映射都稳定后，才允许将筛选同步到路由。

## 反模式

- 把通用筛选组件做成一个巨型业务组件，同时硬编码字段、值编辑器、API 与样式
- 为远程选项只做本地过滤，导致输入关键字后依旧看到全量历史数据
- 只把 id 编进条件和路由，却不提供反查标签逻辑
- 条件变更、URL 变更、搜索请求各自独立触发，造成重复请求或回显错乱
- 为了省事把复杂关联筛选强行映射到普通字符串字段
- 在 Token 化输入框外再套一层独立编辑态，打断连续操作

## 自检清单

- 是否复用了统一的条件模型、编码与回显工具
- 字段、条件和值编辑职责是否清晰分层
- 远程选项是否真正走接口过滤与按值回查
- Token 展示值与接口提交值是否已分离
- 快捷筛选、保存搜索、主筛选是否共用同一份 `terms`
- 路由回显后，是否能完全还原用户可见名称和条件语义
- 是否避免了重复请求、重复搜索和切换即丢条件的问题
