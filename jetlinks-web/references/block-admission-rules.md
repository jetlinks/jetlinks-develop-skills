# JetLinks Web Block Admission Rules

本文件用于判断一个页面区块是否值得存在，尤其是统计卡、图表、概览区、快捷入口、推荐操作区。

## 区块准入四问

任何区块在加入页面前，都要回答这四个问题：

1. 给谁看？
2. 帮助做什么决策或动作？
3. 数据从哪里来？
4. 看完之后下一步是什么？

有任一问题答不上来，这个区块通常就不该出现。

## 常见区块判断

### 原型说明 / 开发备注区

保留条件：
- 仅存在于线框图、草图、设计评审稿中，用于团队对齐

删除条件：
- 进入正式实现后仍以页面文案、区块标题、说明卡片、占位标签等形式保留

原则：
- 终端用户不需要看到“交互方式”“模块原理”“这里以后接数据”“设计意图”这类说明
- 如果用户真的需要帮助，应改写为面向使用的提示、帮助文档入口或上下文化说明

### 统计卡 / KPI 卡

保留条件：
- 数字能直接帮助判断风险、优先级、进度或结果
- 用户会因为这个数字改变操作顺序或处理策略
- 已确认真实数据源、刷新方式和后续动作承接

删除条件：
- 只是“看起来像首页应该有”
- 没有真实来源
- 数据源、刷新方式或业务用途说不清
- 即使删掉也不影响用户判断和动作

### 趋势图 / 分布图

保留条件：
- 趋势本身就是业务判断依据
- 用户需要观察变化，而不是只看当前值
- 已确认真实数据源与钻取或后续动作

删除条件：
- 只是为了让页面更丰富
- 没有后续钻取或动作承接
- 数据只是占位想象，未确认真实来源

### 快捷入口 / 快捷操作

保留条件：
- 是高频动作
- 能显著缩短关键任务路径

删除条件：
- 低频动作被硬塞到首屏
- 与当前页面主任务弱相关

### 概览区 / 摘要区

保留条件：
- 能帮助用户在几秒内理解对象状态或当前情境

删除条件：
- 摘要信息和下方内容重复
- 只是把字段重新排一遍

## 反填满原则

不要因为页面空，就自动添加：

- 装饰性统计数字
- 假趋势图
- 无来源排行榜
- 没有动作承接的概览卡
- 与当前任务无关的运营数据

页面“空”不一定是问题；无价值的信息占空间才是问题。

## 最终检查

在交付前再问一次：

- 如果把这个区块删掉，用户会失去什么？
- 这个区块是否缩短了任务路径、提升了判断效率，或降低了出错概率？
- 这个区块是否值得首屏空间？

如果答案都不明显，就删掉或降级到次级区域。
