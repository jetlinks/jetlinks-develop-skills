# JetLinks Web Page Pattern Decision Rules

本文件用于在“列表页 / 工作台 / 详情页 / 分步流 / 看板 / 概览页”等页面分型之间做选择。

## 决策顺序

1. 先判断用户进入页面后的第一任务是什么。
2. 再判断用户操作的是“一批对象”还是“一个对象”。
3. 再判断关键决策依赖的是“筛选检索”还是“状态理解 / 处置推进 / 关系分析”。
4. 最后才决定是否需要列表、卡片、图表、步骤条、时间线、侧边详情等结构。

## 页面分型

### 1. 标准 CRUD 管理页

适用：
- 核心任务是批量检索、筛选、增删改查
- 用户主要面对的是一批同类对象
- 决策主要依赖表格字段，而不是复杂状态流

推荐结构：
- 搜索区
- 表格或卡片列表
- 批量操作 / 行操作
- 新增 / 编辑弹窗或抽屉

避免：
- 为了显得丰富而额外堆概览卡、趋势图、排行榜

### 2. 对象详情驱动页

适用：
- 用户主要围绕单个对象持续查看、编辑、排查、处置
- 关键问题是“这个对象现在怎么样、下一步怎么做”

推荐结构：
- 顶部对象摘要
- Tabs / 分区详情
- 相关记录、事件、日志、配置
- 侧边动作或固定操作区

避免：
- 先展示一堆与当前对象无关的全局指标

### 3. 工作台 / 处置台

适用：
- 用户进入页面是为了快速识别问题并执行动作
- 页面要同时承载待办、状态、风险、最近变化、快捷操作

推荐结构：
- 优先级信息区
- 待处理事项 / 异常区
- 关键对象列表
- 快捷入口与联动动作

避免：
- 没有动作承接的静态数据墙

### 4. 概览 / 监控页

适用：
- 用户核心任务是理解整体运行状态、趋势和异常
- 页面更像“感知与判断”，不是直接录入和编辑

推荐结构：
- 真正有意义的状态指标
- 趋势或分布图
- 异常聚焦区
- 钻取入口

避免：
- 没有来源和决策价值的数字卡
- 与监控任务无关的 CRUD 表单区

### 5. 分步流 / 向导页

适用：
- 用户必须按阶段完成配置、开通、发布、联动、审批等流程
- 阶段顺序与校验关系明显

推荐结构：
- Steps / 分段导航
- 阶段性表单或配置区
- 阶段结果与下一步动作

避免：
- 把强顺序流程压扁成一页杂糅表单

### 6. 时间线 / 记录分析页

适用：
- 用户需要理解事件演进、操作历史、状态变化
- 决策依赖“先发生什么、后发生什么、为何变化”

推荐结构：
- Timeline / 日志流
- 过滤与定位
- 关键节点高亮
- 节点详情

避免：
- 用普通表格替代需要时序理解的场景

## 选择校验

在确定分型前，至少回答：

- 这个页面最重要的一个任务是什么？
- 用户更像在“找对象”，还是在“处理对象”？
- 用户是否需要快速执行动作，而不仅仅是看数据？
- 如果删掉列表、图表、统计卡中的一个，核心任务还能否完成？

硬触发：

- 如果“目标用户、第一任务、成功标准、单对象/多对象判断”中有 2 项及以上说不清，必须先问用户。
- 如果标准管理页、详情页、工作台、概览页之间仍无法收敛到一个主模式，必须先问用户。

如果回答后仍不明确，先出线框图再实现。
