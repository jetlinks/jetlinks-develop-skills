# JetLinks 根因处理与禁用规避手法规则

本文件适用于所有 JetLinks 后端任务（CRUD、boundary、events、reactive、protocol、conventions）。当框架、SDK、第三方库或现有 API 不直接满足需求时，必须从根本上解决问题，禁止用"奇技淫巧"绕过类型系统、可见性、生命周期或错误传播。

## 适用场景

命中以下任意一项时，本规则强制生效：

- 无法直接访问目标类 / 方法 / 字段（private、package-private、protected、final、sealed、模块封装）
- 序列化 / 反序列化报错（Jackson、Protobuf、Hessian、自定义编解码等）
- 响应式链路报错或需要等待结果（`Mono` / `Flux` 与阻塞 API 互转）
- 类型不兼容、泛型擦除、协变/逆变、运行时类型推断失败
- 第三方 SDK / 库 / 框架 API 行为不符合预期或缺少所需扩展点
- 异常体系不直接支持当前场景（缺少业务异常类、缺少 i18nCode 模型）
- 编译告警 / Lint / 类型检查报错
- 单元测试或集成测试因脚手架数据不全、时序不稳、第三方依赖不可用而失败
- 接口契约与实际实现不一致（schema、Topic、Command、事件 payload 缺字段）

## 红线核心原则

1. **优先找官方 / 框架扩展点**：先确认目标库是否提供等价公共 API、SPI、扩展接口、Module 注册、Bean 后置处理器或事件钩子。
2. **次选相邻模块统一封装**：如果当前 workspace 已有适配层、工具类、装饰器或事件机制承接同类问题，先沿用，不要造平行实现。
3. **再考虑版本 / 库的选择**：如果根因在依赖本身，先评估升级版本、换稳定库、或正确配置；变更后果与用户对齐。
4. **不能解决就告知用户**：当上述路径都不可行时，明确告诉用户问题边界、可选方案、风险和建议，让用户选择是否接受降级或绕行，不在代码里偷偷塞规避方案。
5. **任何反射 / Unsafe / 改可见性 / 复制源码 / monkey patch / 字节码注入 / 改类加载器** 的方案，必须**先告知用户并取得确认**，不允许在交付里默默使用。

## 禁止的奇技淫巧（按场景）

### 无法访问目标方法 / 字段

禁止：

- 反射强制调用 + `setAccessible(true)` 绕过可见性
- `sun.misc.Unsafe`、`MethodHandles.privateLookupIn` 等绕过封装的写法
- 复制粘贴第三方源码到本地修改后再使用
- 把自己的代码挪进库的包名下，以 package-private 身份访问
- 用 cglib / javassist / ASM 在运行时改字节码绕过限制
- 在自己代码里手写一个同名同签名的"伪"类骗类型系统

必须做：

- 找官方公共 API、SPI、扩展点；查源码确认是否有 `ExtensionPoint`、`Customizer`、`Builder.Hook`、`@Configuration` 或事件钩子
- 通过组合 / 装饰 / 代理 / 命令边界 / 事件订阅完成需求
- 如果只能通过私有 API 达成，先告知用户后果（版本升级会破裂、不被官方支持）并取得确认

### 序列化 / 反序列化报错

禁止：

- 把字段 `toString()` 后塞 `String`，绕过结构化序列化
- 改字段类型为 `Object` / `Map<String,Object>` 避开类型推断
- 用 `@JsonIgnore` / `transient` 把**业务必需字段**忽略掉
- `catch(JsonProcessingException e){}` 静默吞掉
- 自己实现一套 `ObjectMapper` 复制配置而不复用模块标准配置
- 把对象先转 JSON 字符串再当字段塞进另一个对象

必须做：

- 定位真实失败原因：自定义类缺空构造、循环引用、接口 / 抽象类未告知具体类型、泛型擦除、字段访问器缺失
- 用框架标准方案：自定义 `JsonSerializer` / `JsonDeserializer`、`@JsonTypeInfo` + `@JsonSubTypes`、`Mixin`、`Module` 注册
- 跟随相邻模块的序列化约定与 ObjectMapper 配置
- 如该字段在当前协议下确实不能直接序列化，告知用户并给出可选路径（拆字段、引专属编解码、走二进制）

### 响应式链路与异步

禁止：

- 在响应式业务流中直接 `block()` 跳出，除非相邻模块同边界已有先例
- 在 `Mono` / `Flux` 中嵌套 `subscribe()` 控制主业务流
- 把阻塞调用包进 `Mono.fromCallable` 后声称是非阻塞
- `CompletableFuture.get(...)` 同步等待响应式结果
- 用 `Thread.sleep` 等待异步副作用完成
- 用全局可变 `static` 状态绕过响应式上下文传递

必须做：

- 按 `$jetlinks-reactive` 规范保持 `Mono` / `Flux` 端到端
- 阻塞 IO 必须用模块已有的 Scheduler 隔离，并明确告知用户
- 时序协调走 `flatMap`、`zip`、`then`、`concatMap`、`Sinks`、命令 / 事件，不靠 sleep
- 异步副作用要么订阅入主链路，要么走事件 / 命令模型

### 类型不兼容 / 泛型

禁止：

- `@SuppressWarnings("unchecked")` 直接强转盖问题
- 反射 `Field.set` 改 final / private 字段
- 改成 `Object` / 原始类型避开编译错误
- 自定义 `equals` / `hashCode` 临时绕开类型差异

必须做：

- 复用相邻模块的转换工具或 `Converters`
- 确认接口契约后用泛型边界 / `TypeRef` / `ParameterizedTypeReference` 明确类型
- 真不兼容时告知用户并提议改契约或新增适配层

### 第三方 API 行为不符合预期

禁止：

- 反射改私有字段或私有方法
- 把库代码拷到本地修改后再编译进项目
- 用 AOP / 字节码注入静默替换库行为
- 给依赖打 monkey patch 后不留迹象

必须做：

- 先确认是否使用方式不当（配置、初始化顺序、线程模型）
- 看官方扩展点 / 升级版本 / 找有维护的替代库
- 通过适配层 / 装饰器 / 事件订阅承接差异
- 实在不行告知用户，让用户决定是否接受用反射 / monkey patch 的方案

### 异常处理

禁止：

- `catch(Exception e) {}` 静默吞掉
- `catch(Exception e) { e.printStackTrace(); }` 当作处理
- 用 `try/catch` 把整段业务包起来掩盖具体失败点
- 把检查异常转成 `RuntimeException` 直接抛出，不带业务语义
- 在异常构造里硬编码中文 / 英文 message（违反 i18n 约定）

必须做：

- 让异常按既定层级传播，或转换成 `i18nCode` / message key 的业务异常
- 在边界层处理失败，给出具体业务语义
- 失败原因关键时进结构化日志（带对象 ID、操作、上下文），不靠 `printStackTrace`

### 编译告警 / Lint / 类型检查

禁止：

- 大范围 `@SuppressWarnings` 把告警一删了之
- 把 strict mode 关掉绕过检查
- 删测试 / 改断言来"通过"测试
- 注释掉编译失败的代码完成"交付"

必须做：

- 修真实问题；只对确实合理且无解的告警局部 suppress 并加注释说明原因

### 测试 / 数据 / 时序

禁止：

- 在测试里 `Thread.sleep` 等异步副作用，不写明上限和告警
- mock 返回固定值掩盖真实 bug
- 改测试期望以适应错误的实现
- 注释掉失败的测试用例

必须做：

- 用 `StepVerifier`、`Awaitility`、`TestPublisher` 等正确等待异步
- mock 行为要忠于真实契约
- 失败用例先定位代码或测试假设错在哪

## 必须告知用户的触发条件

满足以下任一条件，**实现前必须告知用户并等待确认**，不允许直接交付：

- 需要使用反射 / Unsafe / 改可见性 / 复制源码 / 字节码注入 / monkey patch / 改类加载器
- 需要让响应式模块出现 `block()` 或同等阻塞调用
- 需要新增依赖（特别是版本不稳定 / 维护停滞的库）
- 需要修改第三方库的行为或版本
- 需要绕过框架既定的扩展点 / 边界 / 事件机制
- 需要在 i18n 约定的模块里硬编码中文 / 英文 message
- 业务必需字段无法被现有协议或序列化方案表达
- 测试或验证在当前环境无法完成（环境隔离、依赖缺失、数据不齐）

告知模板（要素必备）：

- 现有方案为什么不可行（具体证据：API 签名、文档、源码、报错）
- 可选路径 A / B / C 各自的代价和风险
- 推荐路径与理由
- 如果用户选择最小代价但带风险的路径，需要在代码或交付说明里留痕

## 自检清单

- 是否定位到了根因（不是表象）？
- 是否优先看了官方扩展点、相邻模块封装、版本与库选择？
- 任何反射 / Unsafe / 改可见性 / 复制源码 / monkey patch 是否已告知用户并取得确认？
- 异常是否按规范传播 / 转业务异常，没有被静默吞？
- 响应式模块是否端到端保持非阻塞？
- 编译告警与失败测试是否真修，而不是被压制 / 注释？
- 如果做不到根因解决，是否已经明确告知用户问题边界、可选方案与建议？
