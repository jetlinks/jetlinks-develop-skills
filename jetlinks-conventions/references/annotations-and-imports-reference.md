# JetLinks 注解与导入确认指南

本文件不是固定包名清单，而是帮助智能体在不同 JetLinks 系脚手架、不同 Spring/HSWeb 版本下准确确认导入。

## 核心原则

1. 先看相邻代码，再写导入
   - 同一个脚手架里可能同时存在 `javax` 与 `jakarta` 迁移痕迹。
   - 同类文件的现有导入永远比记忆可靠。

2. 先确认注解家族，再确认具体包
   - 例如先确认“实体表注解用什么”，再确认具体是 `javax.persistence.Table` 还是其他。

3. 不要因为文档示例直接复制导入
   - 示例只表达模式，不表达你当前工作区的最终包路径。

## 常见确认项

### 实体层

确认：
- 表映射注解
- 字段列注解
- 枚举、JSON、默认值、ID 生成相关注解
- 是否启用实体事件
- 审计字段接口或基类

### 服务层

确认：
- 服务基类或接口
- 是否使用响应式 CRUD 基类或阻塞式基类
- 是否需要事务、日志、校验

### 控制器层

确认：
- REST 注解
- 权限注解
- Swagger/OpenAPI 注解
- CRUD 控制器基类或接口

### 命令服务

确认：
- `@CommandService`
- 命令处理注解
- 参数选择器、查询组件、权限注解

### 订阅与事件

确认：
- `@Subscribe`
- `@EventListener`
- 事务事件监听或集群事件抽象

## 推荐检查顺序

1. 找目标模块中最像的现有类
2. 复制其导入家族
3. 只替换业务类型和方法签名
4. 再补充当前功能独有的注解

## 高风险项

这些内容最容易因版本差异写错，必须先确认：

- `javax.persistence` / `jakarta.persistence`
- `javax.validation` / `jakarta.validation`
- OpenAPI/Swagger 注解包
- 事务相关注解包
- CRUD 控制器和服务基类的具体包路径

## 输出模板

当用户要求先确认导入时，输出：

1. 需要确认的注解家族
2. 已参考的相邻文件
3. 最终采用的包路径
4. 是否存在版本混用风险
