# JetLinks 国际化规则

本文件用于判断当前 JetLinks 模块是否应该补国际化，以及补国际化时必须遵守哪些
边界。只有在确认模块已有 i18n 约定，或用户明确要求新增时，才继续深入实现。

需要具体实现示例时，再读 [`i18n-usage.md`](i18n-usage.md)。

## 第一原则：先确认当前模块是否真的使用 i18n

检查：

- 是否存在 `src/main/resources/i18n/...`
- 是否存在 `messages_zh.properties`
- 是否存在 `messages_en.properties`
- 相邻模块是否对权限、动作、枚举、字段、错误消息做了本地化

如果当前脚手架没有这套约定，不要为了“形式完整”而臆造新的资源路径。只有用户
明确要求，或你正在创建一个明确需要多语言支持的新模块时，才引入新的 i18n 目录。

## 何时需要补 i18n

- 新增枚举且前端会直接展示其文本
- 新增实体字段且当前模块已有字段国际化惯例
- 新增 `@Resource` 权限 ID 且当前模块已有权限国际化
- 新增 `@ResourceAction` 或等价动作 ID 且当前模块已有动作国际化
- 新增用户可见错误、提示、告警或状态消息

通常不需要补 i18n 的内容：

- 纯内部日志
- 调试信息
- 只给开发者看的异常堆栈细节
- 当前模块完全没有国际化约定且用户未要求新增

## 常见键模式

以下是常见模式，但是否采用以当前模块现状为准：

- 枚举：`完整包名.枚举类名.枚举值`
- 实体字段：`完整包名.实体类名.字段名`
- 权限：`hswebframework.web.system.permission.{权限ID}`
- 动作：`hswebframework.web.system.action.{动作ID}`
- 错误：`error.{key}`
- 提示：`message.{key}`

## 编写规则

1. 键名和代码中的标识保持一一对应。
2. 模块已有 `messages_zh.properties` 和 `messages_en.properties` 时，尽量同步补齐。
3. 如果相邻模块只维护一种语言，跟随现有惯例。
4. 不要为内部日志、调试信息补国际化。
5. 不要手动传 `LocaleUtils.current()` 给 `resolveMessage(...)`，除非本地代码已经这样做。
6. 不要在同一模块里混用 UTF-8 中文和 `\uXXXX` 转义，除非现有文件已经如此。
7. 对 `I18nEnumDict`，key 必须与完整包名、类名、枚举值完全一致。

## 注意事项

- 当前工作区可能存在“已有权限注解但未完全补 i18n”的历史情况。
- 因此规则不能写成绝对强制，而应要求先判断模块习惯，再补一致的内容。
- 代码里仍应优先保留默认文案，不要让资源缺失直接导致空白或不可理解的提示。

## 自检清单

- 当前模块是否已有 i18n 目录或明确的 i18n 惯例
- 这类键在相邻模块中是否已被使用
- 新增的权限、动作、枚举、字段、错误码是否与代码标识一致
- 多语言文件是否保持同步
- 新增文本是否真的是用户可见内容
