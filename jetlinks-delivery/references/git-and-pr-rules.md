# Git、分支、PR 与测试交付规则

当任务不仅涉及“写代码”，还涉及“提交代码、准备交付、发起 PR、说明测试结果”时，读取本文件。

## 目标

确保 JetLinks 项目的交付过程符合以下要求：

- 提交标题对齐仓库既有风格
- 禁止直接 push 到主干或集成版本分支
- 通过 PR 合并
- 提交前完成单元测试或集成测试
- 在 PR 描述中用数据说明测试结果和覆盖率

## 提交标题规范

优先对齐仓库现有历史，默认采用：

```text
type(scope): summary
```

约束：

- `type` 与仓库历史保持一致，优先使用 `feat`、`fix`、`refactor`、`docs`、`test`。
- `scope` 选择本次改动的真实模块、业务域或规则域。
- `summary` 使用简洁中文动宾短语，直接说明结果。
- 除非历史明确允许，否则不要省略 `scope`。

参考仓库历史风格：

- `refactor(prompt): 扩展低上下文边界决策规则`
- `refactor(基础模块): 优化菜单逻辑`
- `refactor(设备管理): 优化实体拓展型`

避免：

- `update`
- `fix bug`
- `misc`
- `test`
- 无法体现模块和结果的空泛标题

## 分支与合并策略

禁止直接 push 到以下分支：

- `master`
- `main`
- 版本集成分支，例如 `2.11`、`2.12`
- 其他被团队视为主线或发布线的长期分支

标准流程：

1. 确认目标合入分支，例如 `2.12`。
2. 从该基线分支拉出临时开发分支。
3. 在临时分支上完成修改、测试和提交。
4. push 临时分支到远端。
5. 创建 PR，base 指向目标版本分支。

如果用户要求“直接推到 2.12”这类操作，应改为：

- 说明该分支属于受保护主线，不直接 push。
- 改走“临时分支 + PR”流程。

## GitHub PR 创建方式

如果当前仓库远端是 GitHub，且用户要求“发 PR”“提交 PR”“推到 GitHub 并开 PR”，默认推荐使用 `gh` 创建 PR，而不是只给一个网页链接让用户手动补完。

推荐顺序：

1. 用 `gh auth status` 检查 GitHub CLI 是否已登录。
2. 用 `gh repo view` 确认仓库、默认分支和目标 base。
3. 在临时分支 push 完成后，优先使用 `gh pr create` 创建 PR。
4. 除非用户明确要求直接进入 review，默认创建 draft PR。

带沙箱的 agent 环境额外约束：

- `gh` 相关命令不要在沙箱内执行。
- 包括 `gh auth status`、`gh repo view`、`gh pr create` 在内，都应申请非沙箱执行。
- 如果认证失效，先明确阻塞点，让用户完成 `gh auth login` 后再继续。

允许的降级方式：

- 如果 `gh` 未安装，先说明阻塞原因。
- 如果用户不希望 agent 直接创建 PR，再退回到提供可点击的 PR 链接。
- 只有在 `gh` 不可用或用户明确要求手动处理时，才不使用 `gh pr create`。

## 测试要求

提交前必须执行自动化测试，至少覆盖本次改动涉及的核心路径。

优先级：

1. 先跑与改动模块直接相关的单元测试。
2. 如果改动跨模块、跨边界、跨服务或事件链路，再补集成测试。
3. 如果仓库已有覆盖率门槛，必须满足门槛后再准备 PR。

最低交付要求：

- 给出测试命令
- 给出通过数、失败数、跳过数
- 给出覆盖率结果，或说明项目缺乏覆盖率工具时的替代证据

可接受的证据示例：

- `mvn -pl modules/device-manager -am test`
- `mvn -pl modules/device-manager -am verify -P integration-test`
- surefire / failsafe 报告中的 passed、failed、skipped
- jacoco 报告中的 line / branch 覆盖率

不合格的说法：

- “已测试”
- “本地没问题”
- “理论上可用”

如果测试受阻：

- 明确阻塞原因，例如环境依赖、外部服务、数据库、消息中间件
- 明确已完成的替代验证
- 明确仍然存在的风险
- 未满足团队要求前，不把该提交描述为可合并状态

## PR 描述要求

PR 描述必须是结果导向，不写空话。

至少包含：

1. 目的
2. 核心变动
3. 测试结果
4. 风险与未覆盖项

推荐模板：

```md
## 目的

- 修复 / 优化 / 新增什么能力
- 解决什么业务或技术问题

## 核心变动

- 模块 A：改动点
- 模块 B：改动点

## 测试结果

- 命令：`mvn -pl xxx -am test`
- 单元测试：42 passed, 0 failed, 1 skipped
- 集成测试：8 passed, 0 failed
- 覆盖率：line 81.4%, branch 73.2%

## 风险与说明

- 影响范围：
- 未覆盖场景：
- 已知限制：
```

要求：

- 测试结果必须有数字
- 覆盖率必须有数字，或明确写出缺失原因与替代证据
- 不要只写“测试通过”

## 执行提示

当用户要求“提交代码”“推送代码”“发 PR”“整理交付说明”时：

1. 先检查当前分支是否为主干或版本分支。
2. 如果是主干或版本分支，不直接 push，先创建临时分支。
3. 根据仓库历史整理合规的 commit title。
4. 运行与改动相关的单元测试或集成测试。
5. 收集测试通过数、失败数、跳过数和覆盖率结果。
6. 如果远端是 GitHub，先检查 `gh auth status`，并在带沙箱环境中申请非沙箱执行。
7. 生成 PR 标题与 PR 描述，确保包含目的、核心变动、测试数据和风险说明。
8. push 临时分支后，优先使用 `gh pr create` 创建 draft PR；如果 `gh` 不可用，再明确说明阻塞或降级方式。
